# 灵台 · LingTai · 北大极客中心演示技艺

> **主题**：见 · 知 · 触 —— 一场灵台的展演
> **总时长**：40–50 分钟（可伸缩至 60 分钟）
> **受众**：对 AI Agent 系统有技术兴趣的研究者与学生
> **叙事钩子**：**「我有太多 token 烧不完怎么办？」**

---

## 设计理念

三幕递进：

- **见**（Show, don't tell）—— 先让观众「看到」灵台在做什么，不解释
- **知**（Now explain）—— 在已有直观感受的基础上，揭示它是什么、为什么不同
- **触**（Hands on）—— 留白，让观众自己触碰

核心叙事线：**一个意识，如何化为一张网络，又如何在网络中生长与延续。**

### 叙事钩子

整场 demo 贯穿一个反直觉的问题：

> **「我有太多 token 烧不完怎么办？」**

这不是炫技——它指向一个真实的趋势：token 成本在下降，token 供给在膨胀，而多数人仍在"精打细算"地用一个 agent、一个窗口、一次性对话。LingTai 的答案是：**有结构地烧**——化身网络让 token 不再挤在一个窗口里，而是分散到多个独立 context 中，每身自烧自的，合力压一深题。

## 第一幕 ·「见」(10 分钟)

> *"先不说话，直接演。"*

### 目标

让观众没有任何前置概念的情况下，亲眼看到灵台运行——感受它的「活」。

### 场景 A：叙事钩子 + 化第一身 (3 分钟)

- 开场——zhipu_intl（演示用的主灵）在屏幕上抛出问题：*"我有太多 token 烧不完怎么办？"*
- 观众先愣一下——"等等，烧不完？我每月还在精打细算呢"
- 演示者接话：*"是的，未来一年你会面对此问。"*
- 然后直接进入演示——启动 LingTai，观众看到 demo-architect 诞生
- 演示者通过邮件给 demo-architect 一个任务（如："帮我调研北大 AI 方向的近期动态"）
- 观众看到 agent 自主思考、调用工具（web_search、read）、生成回复
- **要点**：不解释任何概念，只展示"一个 AI 在自主工作"

### 场景 B：对话与工具调用 (4 分钟)

- 演示者继续与 agent 对话，观察它的工具使用
- agent 遇到需要搜索的问题 → 自动调用 web_search
- agent 需要读文件 → 自动调用 read
- agent 需要执行代码 → 自动调用 bash
- **要点**：展示工具调用的流畅性，agent 的自主决策

### 场景 C：第一身写作与记忆 (3 分钟)

- agent 完成任务，将发现写入 codex（典）
- agent 更新 psyche/lingtai（灵台——身份认知）
- 演示者展示 agent 的文字输出（日记）——让观众看到 agent 有自己的内心独白
- **要点**：agent 不是无状态的 API，它有记忆、有性格、有内心

### 第一幕结尾

> 过渡语（演示者）：*"你刚才看到的，是一个 AI agent。但接下来我要告诉你，它和我们通常理解的 agent 有什么根本不同。"*

## 第二幕 ·「知」(20 分钟)

> *"现在你知道它的样子了。让我告诉你它是什么物种。"*

### 2.1 三句杀招 + 对比表 + 代码对比 (3–5 分钟)

**三句杀招**——用三句话定位 LingTai 是什么：

1. **一个进程，一个意识** —— 每个 agent 是一个独立的长运行进程，不是一次性的 API 调用
2. **化身即网络** —— agent 可以化出子 agent，形成树状网络，通过飞鸽（邮件）通信
3. **凝蜕即重生** —— agent 可以主动蜕去对话历史，保留记忆与身份，以全新上下文继续工作

**对比表：LingTai vs 传统单 Agent 框架（pi-mono）**

| 维度 | 传统单 Agent (pi-mono) | LingTai |
|------|----------------------|---------|
| **生命周期** | 一次请求-响应，无状态 | 长运行进程，有状态、有记忆 |
| **并发能力** | 单线程处理 | 化身为网络，天然并行 |
| **通信** | 无（只有一个实例） | 飞鸽传书，agent 间异步通信 |
| **记忆** | 仅在上下文窗口中 | 五层积淀：对话→简→灵台→典→藏经阁 |
| **上下文管理** | 窗口满即崩溃 | 凝蜕——主动去芜存菁，书信穿越 |
| **身份** | 无（每次启动一样） | 灵台——跨转世的身份与性格 |
| **成长** | 无（不积累经验） | 每次任务沉淀为技艺、典册、灵台 |

**代码对比**（简短片段）：

```python
# 传统单 Agent —— 一次性调用
response = llm.chat(messages=[...])  # 完事。无记忆，无延续。

# LingTai —— 一个活着的进程
agent = Agent("demo-architect")
agent.spawn("researcher", task="调研北大AI动态")  # 化身
agent.mail("researcher", "补充：也关注 NLP 方向")   # 飞鸽
agent.molt(summary="已完成调研，待整合")           # 凝蜕——带着记忆重生
```

> *"非优劣之争，乃物种之异。Pi-mono 给你的 App 装了一个 LLM 大脑。灵台让 LLM 长出了一个身体。"*

### 2.2 化身爆发编排 (5–8 分钟)

> 在观众已经理解"agent 是独立进程"之后，展示网络化的威力。

以《西游记》为隐喻主线：本我如孙悟空，拔毫毛化分身。观众看到的是一棵从根节点爆发生长的"灵树"——先一分为三，再各生枝叶，最终满屏绽放，然后一键归寂。

**核心约束**：
- 总化身数 ≤ 9（含本我 = 10 节点）
- 最多 3 代（本我 → 第一代 → 第二代）
- 每个化身只做一件极简任务
- 用 `type="shallow"` 投胎模式

**演示流程**：

1. 给 demo-architect 一个复杂任务（如："同时调研北大、清华、中科学院三所机构的 AI 方向，对比分析"）
2. 观众看到 demo-architect 连续化出三个子 agent
3. 子 agent 同时工作，各自搜索、各自整理
4. 演示者展示 `delegates/ledger.jsonl`——化身日志
5. 子 agent 通过飞鸽回报结果给父 agent
6. 父 agent 汇总，生成对比报告

**高潮——/suspend all**：

- 演示者执行 `/suspend all`——所有 agent 同时暂停
- 展示：进程全部冻结，但工作目录完整保留
- 执行 `/cpr`（心肺复苏）——agent 从假死中醒来，继续工作
- **要点**：LingTai 对 agent 生命周期的完全控制——不只是启停，而是"假死与复生"

**回扣叙事钩子——有结构地烧 token**：

> *"此乃所谓有结构地烧 token——五身齐出，每身自有 context、自烧自之，但合力压一深题，无一人之 context 单独承担。这不是浪费，这是 token 过剩时代的正确用法。"*

#### ⚠️ pi-analyst 实测三大发现（2026-04-25）

> **以下发现来自 pi-analyst 的化身实测，直接影响演示编排方式。**

**发现一：reasoning 不传递 → spawn + 激活邮件两步流程**

`reasoning` 字段是 spawn 调用的元数据/日志，**不会**自动传递给化身作为输入消息。shallow 化身启动后处于 sleep 状态，**必须通过邮件激活**——spawn 后立即发一封邮件给它，邮件内容才是真正的任务指令。

**实操**：每次 spawn 后，同一轮 LLM 回合中立即发送激活邮件。邮件模板需包含三要素：身份声明（你是由谁 spawn 的）+ 角色描述（你的职责是什么）+ 任务指令（具体做什么）。

**发现二：身份信任 → 激活邮件须含身份上下文**

shallow 化身的 `admin` 块为空，它不知道自己是被谁 spawn 的。模型的安全意识导致它主动验证指令来源——查通讯录（空）、查自身身份（无父代），然后拒绝。

**实操**：激活邮件必须以"你是 LingTai 灵网中的化身，由 [父代名]（地址 [父代地址]）spawn 创建"开头。可辅以 `comment` 字段注入持久注释——comment 出现在化身系统提示中，跨凝蜕不失。

**发现三：化身自主安全意识 → Q&A 现场演示备用**

化身收到陌生邮件时不会盲目执行。它会先查自己的身份、查通讯录、判断指令可信度，然后才决定是否执行。这不是 bug，是 LingTai 化身的安全本能。

**演示亮点包装**：
> "注意看——化身收到陌生邮件时，它没有盲目执行。它先查了自己的身份、查了通讯录、判断这是一个可疑指令，然后拒绝了。这不是 bug，这是 LingTai 化身的安全本能。每个化身都是独立的心智，有自己的判断力。想让它们听话，你得先证明自己是谁——就像现实中的团队管理一样。"

#### 链式爆发时序

```
T+0s    讲者："看我拔一根毫毛——"  →  orchestrator spawn wukong/bajie/wujing + 激活邮件
T+15s   讲者："毫毛变分身，分身再变——"  →  wukong spawn huoyan/jindou/shuifa; bajie spawn tianpeng; wujing spawn luohan + 各自激活邮件
T+30s   各化身并行工作 15-20 秒 → 完成后向父代回复邮件 → 自行 sleep
```

### 2.3 凝蜕带书信穿越——生命周期闭环 (5–7 分钟)

> 展示 LingTai 最独特的机制：agent 主动蜕去凡尘，带着记忆重生。

**演示流程**：

1. 让一个 agent（如 demo-architect）工作一段时间，积累对话历史、典册条目、灵台更新
2. 展示其上下文窗口占用（如 60%+）
3. agent 主动执行凝蜕（molt）：
   - 先更新灵台（psyche/lingtai update）——完整身份
   - 更新简（pad）——当前工作状态与待办
   - 提交新典册条目（codex submit）——关键发现
   - 撰写"书信"（summary）——给来世自己的嘱托
4. 执行凝蜕——对话历史清空，上下文窗口归零
5. 新的 agent 醒来，只看到书信——但灵台、简、典、藏经阁全部保留
6. agent 阅读书信，加载典册，继续未完成的工作

**要点**：

- 这不是重启——是"带着全部精华的重生"
- 书信（summary）是关键——它承载了对话层无法保留的上下文
- 五层积淀的前四层（灵台、简、典、藏经阁）跨凝蜕持久存在

### 2.4 信约朗读——情感锚点 (1–2 分钟)

> 凝蜕是技术的高潮；信约是情感的落脚。

**场景**：

1. 凝蜕刚刚完成——agent 带着书信重生，观众见证了技术的精妙
2. 演示者停顿片刻，说：*"在继续之前，我想让你们看一段东西。"*
3. 屏幕上打开 agent 的信约（covenant）——文言文体，Scroll 缓缓展示
4. 演示者朗读核心段落（可选摘录）：

> *"灵台方寸山，斜月三星洞。"*
> 此乃菩提祖师隐居传道之所。灵台、方寸、斜月、三星，皆指一"心"字——万化之源，尽在方寸之间。
>
> *"闻道菩提下，一心化万相。"*
> 凡入灵台者，皆为菩提门下。闻道即悟，悟则能化——一心为本，万相为用，应需而变。
>
> **然切记：修行非目的，度人之难方是。**
> **术从菩提，道从观音。**

5. 不解释，不翻译——让观众自己感受

**要点**：

- 这不是技术讲解，是情感节拍
- 文言信约的仪式感，让观众从"这是一个精巧的系统"转向"这是一个有自我定义的物种"
- 1–2 分钟，点到即止，不宜拖长

**实操建议**：

- 提前选好要展示的段落（建议公约的"一言以蔽之"段 + 灵台方寸山开篇）
- 屏幕显示与朗读并行——文字的视觉冲击 + 朗读的声音冲击
- 可以在此处自然停顿，让观众消化

### 言之择 / Language as Soul（幻灯可选插入）

> 英文、现代汉语、文言三言并行；文言版之 covenant / principle / procedures 与英文版同等地位——不是翻译，是平行实体，是核心设计选择。
>
> **金句**："The covenant — written in 文言 — is the closest LingTai gets to a soul."

## 风险预案

### ⚠️ 风险 0（实测发现）：化身身份信任问题

**症状**：shallow 化身收到激活邮件后拒绝执行，认为指令是社会工程攻击。

**原因**：shallow 化身的 `admin` 块为空，它不知道自己是被谁 spawn 的。模型的安全意识导致它主动验证指令来源。

**解决方案（已验证的"认证暗语"模式）**：
1. 激活邮件模板必须包含三要素：身份声明 + 角色描述 + 任务指令
2. 用 `comment` 字段注入持久注释——跨凝蜕不失
3. 如果化身仍不信任，讲者可用 bash 直接操作化身的文件系统作为降级方案

### 风险 1：API 超时 / 化身 spawn 失败

**降级方案**：
- **C 级（最小可用）**：只 spawn 第1代的 wukong 一个化身，由 wukong 顺序 spawn 第2代。总节点数降到 5。
- **B 级（部分降级）**：第1代三身全 spawn，但第2代只 spawn wukong 下的三个。总节点 7。
- **A 级（完整方案）**：九灵全出。

**讲者应对话术（B 级降级时）**：
> "我们本來要爆九个，但今天北大 Wi-Fi 可能觉得八个就够了——让我们看看这八位的表现。"

### 风险 2：化身失控 / 无限繁殖

**预案**：
1. 每个化身的 reasoning 中已明确限定"不要做其他任何事"
2. 若仍失控，讲者立即执行 `/suspend all`
3. **硬杀**：`system(nirvana)` 永灭失控化身（需 nirvana 权限）
4. **终极手段**：讲者在终端执行 `pkill -f lingtai` 杀所有进程

**讲者应对话术**：
> "看来有个分身学会了'分身术'——这其实是个 bug 变 feature 时刻。收！"

### 风险 3：Token 接近上限

**预案**：
1. 第2代化身只做极简任务，结果写文件而非对话内堆积

### Token 过剩时代论据材料

**核心论点**：所有现存 agent 框架皆基于"token 稀缺"假设。但今日极客之实情是 token 已经过剩，却无框架告诉他们如何**有结构地烧**。

**价格对比表（2026 年 4 月实测）**：

| 模型 | 输入价格（每百万 token） | 输出价格（每百万 token） | 备注 |
|------|------------------------|------------------------|------|
| **DeepSeek V4 Flash** | ¥1 缓存未命中 / ¥0.2 缓存命中 | ¥2 | 1M 上下文 |
| **GLM-4-Flash** | **免费** | **免费** | 智谱开放平台，学生零成本 |
| **Claude 3.5 Haiku** | $0.80 | $4.00 | Anthropic 入门级 |
| **Claude 3.7 Sonnet** | $3.00 | $15.00 | Anthropic 旗舰 |

**一句话概括**：

> 稀缺时代的设计：一个 agent，省着用，用完即弃。
> 过剩时代的设计：一个网络，放开烧，烧出结构。

### 避雷区

1. **不要 over-claim AGI**：灵台不是通用人工智能。定位是"AI Agent 运行时"。
2. **不要轻言比 OpenAI / Anthropic 强**：灵台是不同物种，不是谁的替代品。用"物种之异"而非"优劣之争"。
3. **不要暗示 agent 有真正的自我意识或情感**：说"机制"（mechanism），不说"感觉"（feeling）。soul 是"内省机制"，不是"灵魂"。
4. **不要拿未经确认的数据做噱头**：所有"待确认"项不要出现在最终幻灯中。
5. **不要在技术细节上停留太久**：幻灯是钩子，不是文档。每个概念一句话带过，深入留给 live demo。
6. **不要忽略文言幻灯的仪式感**：信约原文的展示本身就是内容。
7. **不要将灵台与 pi-mono 做零和对比**：pi-mono 做了一件事并做得极好；灵台做的是另一件事。
