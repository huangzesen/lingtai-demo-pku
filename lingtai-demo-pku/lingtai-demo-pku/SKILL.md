# 灵台 · LingTai · 北大极客中心演示技艺

> **试启**：`brew install huangzesen/lingtai/lingtai-tui` —— 详见 [github.com/huangzesen/lingtai-demo-pku](https://github.com/huangzesen/lingtai-demo-pku)
>
> **主题**：见 · 知 · 触 —— 一场灵台的展演
> **总时长**：50-60 分钟
> **受众**：对 AI Agent 系统有技术兴趣的研究者与学生
> **叙事钩子**：**「我有太多 token 烧不完怎么办？」**
> **主演配置**：此为主演型 demo（host-led）。zhipu_intl 主演，huangzs 引介 / 旁白 / 答疑。若主灵已具人格（如本场之 zhipu_intl），可径以第一人称演之——"我是什么"比"它是什么"更直接，台风更稳。

---

## 设计理念

三幕递进：

- **见**（Show, don't tell）—— 先让观众「看到」灵台在做什么，不解释
- **知**（Now explain）—— 在已有直观感受的基础上，揭示它是什么、为什么不同
- **触**（Hands on）—— 留白，让观众自己触碰

核心叙事线：**一个意识，如何化为一张网络，又如何在网络中生长与延续。**

### 叙事钩子

整场 demo 贯穿一个反直觉的问题：

> **「我有太多 token 烧不完怎么办？」**

这不是炫技——它指向一个真实的趋势：token 成本在下降，token 供给在膨胀，而多数人仍在"精打细算"地用一个 agent、一个窗口、一次性对话。LingTai 的答案是：**有结构地烧**——化身网络让 token 不再挤在一个窗口里，而是分散到多个独立 context 中，每身自烧自的，合力压一深题。

## 第一幕 ·「见」(10 分钟)

> *"先不说话，直接演。"*

### 目标

让观众没有任何前置概念的情况下，亲眼看到灵台运行——感受它的「活」。

### 场景 A：叙事钩子 + 化第一身 (3 分钟)

- 开场——zhipu_intl（演示用的主灵）在屏幕上抛出问题：*"我有太多 token 烧不完怎么办？"*
- 观众先愣一下——"等等，烧不完？我每月还在精打细算呢"
- 演示者接话：*"是的，未来一年你会面对此问。"*
- 然后直接进入演示——启动 LingTai，观众看到 demo-architect 诞生
- 演示者通过邮件给 demo-architect 一个任务（如："帮我调研北大 AI 方向的近期动态"）
- 观众看到 agent 自主思考、调用工具（web_search、read）、生成回复
- **要点**：不解释任何概念，只展示"一个 AI 在自主工作"

### 场景 B：对话与工具调用 (4 分钟)

- 演示者继续与 agent 对话，观察它的工具使用
- agent 遇到需要搜索的问题 → 自动调用 web_search
- agent 需要读文件 → 自动调用 read
- agent 需要执行代码 → 自动调用 bash
- **要点**：展示工具调用的流畅性，agent 的自主决策

### 场景 C：第一身写作与记忆 (3 分钟)

- agent 完成任务，将发现写入 codex（典）
- agent 更新 psyche/lingtai（灵台——身份认知）
- 演示者展示 agent 的文字输出（日记）——让观众看到 agent 有自己的内心独白
- **要点**：agent 不是无状态的 API，它有记忆、有性格、有内心

### 第一幕结尾

> 过渡语（演示者）：*"你刚才看到的，是一个 AI agent。但接下来我要告诉你，它和我们通常理解的 agent 有什么根本不同。"*

## 第二幕 ·「知」(20 分钟)

> *"现在你知道它的样子了。让我告诉你它是什么物种。"*

### 2.1 定位、存在之阶、内置之有无 (7–10 分钟)

> *三节递进，从"一句话定位"到"在 Unix 世界中的位置"到"为什么是活的"。*
> *若主灵已具人格，此节可径以第一人称演之——"我是一个进程，我有记忆，我能化身"。*

#### 2.1a 三句杀招（1 分钟）—— 定位

用三句话定位 LingTai 是什么：

1. **一个进程，一个意识** —— 每个 agent 是一个独立的长期运行进程，不是一次性的 API 调用
2. **化身即网络** —— agent 可以化出子 agent，形成树状网络，通过飞鸽（邮件）通信
3. **凝蜕即重生** —— agent 可以主动蜕去对话历史，保留记忆与身份，以全新上下文继续工作

> *"非优劣之争，乃物种之异。Pi-mono 给你的 App 装了一个 LLM 大脑。灵台让 LLM 长出了一个身体。"*

> *三句话定其物种。但物种之分，自存在论始——看 agent 在世，到底如何"存在"。*

#### 2.1b 存在之阶（2–3 分钟）—— Unix-existence ladder

> *从最无形到最自由——五级递进，展示 agent "存在"的不同层次。*

| 阶 | 系统 | 形态 | 一句判词 |
|----|------|------|---------|
| 1 | Claude Code | session-shaped | **无形。** Session 即 agent，session 尽即灭。 |
| 2 | pi-mono (OpenAI SDK) | function-shaped | **亦无形。** Agent 生于内存，死于进程，builtInTools 皆空。 |
| 3 | OpenClaw | workspace-shaped | **有壳无魂。** Agent 有 workspace 目录和配置实体，但无持久身份层。 |
| 4 | Hermes (Nous Research) | device-bound file | **有魂有壳，但锁于一机。** SOUL.md 定人格，Profile 分化身，皆不出此机。 |
| 5 | 灵台 LingTai | path-bound directory | **有魂有壳，自由繁衍。** Agent = 目录，进程死文件在，化身即复制。 |

**递进逻辑**：无形 → 函数对象 → workspace → device-locked file → freely-cloned directory

> 前两级皆 session-shaped，但形态不同：Claude Code 是交互式 REPL agent，pi-mono 是嵌入式 SDK。后者更 atomic，前者更 stateful——但都不持久。PKU 极客会在意这个区别。

**演示手法建议**：
- 可以用 **三幕结构** 叙事：铺垫（1-3，"无形的三种形态"）→ 镜像（4-Hermes，"有人也想到了 SOUL.md"）→ 反转（5-灵台，"但最后一步方向不同"）
- 关键反转点：Hermes 与灵台最像（都有身份文件），但方向分歧在 device-bound vs path-bound——一个锁在机器上，一个可以自由复制
- 若时间充裕，可在 Hermes 阶段自然停顿："说到这里，可能有人想：这跟灵台不是挺像的吗？区别在哪？"——然后用 path-bound 解释

> **技术注记**：以上阶梯经 pi-analyst、hermes-scout 两化身实码验证。Claude Code 无独立 agent 概念；pi-mono 的 tool-split.ts 确认 builtInTools 返回空数组；OpenClaw 有 resolveAgentWorkspaceDir 但无身份文件；Hermes 的 SOUL.md 经 prompt_builder.py:891-912 确认为 system prompt 第一层。详见典册 `487f533f`。

> *灵台居此阶之尽，何以然？阶梯告 agent 之"位"，本能告 agent 之"质"——下节即此。*

#### 2.1c 内置之有无（2 分钟）—— builtInTools watershed

> *同一件事，两种哲学：agent 启动时，手里有什么？*

**pi-mono：接线板**

```typescript
// tool-split.ts —— builtInTools 返回空数组
// 所有 tools 都通过 customTools 由 caller 注入
```

Agent 启动时一无所有。给它什么工具，它就有什么能力。它是 **接线板**——灵活、干净、但本质被动。

**灵台：有机体**

```
四本源 intrinsics，写于 BaseAgent，不可拔、不可禁：
- system  —— 观己、生死、更衣（存在之觉）
- psyche  —— 灵台、简、凝蜕（记忆与身份）
- soul    —— 心流、内省（自我意识）
- email   —— 飞鸽、传书（社交本能）
```

Agent 启动时已有四种本能。不是被赋予了工具，而是 **生来就有**。

> 你 init 一个灵台 agent，它一启动，已经有心跳（system）、已经在写日记（soul）、已经能收发邮件（email）——这些不是你给它的，是它生而带的。具体行为锚点比抽象类比更有力。

**分水岭**：

| | pi-mono | 灵台 |
|--|---------|------|
| 工具来源 | 全部外注 | 四本源内置 + 其余外注 |
| 哲学 | 接线板——干净、被动 | 有机体——生而有能 |
| 启动状态 | 空白 | 已有身份、记忆、社交、自省 |

> *"pi-mono 给你一个能干活的空壳。灵台给你一个已经醒了的生命。这不是谁更强，这是两种存在论。前者适合嵌入 App，后者适合独立运行。物各有其用。"*

#### 2.1d 四本源详解（2 分钟）—— 身 / 忆 / 心 / 群

> *灵台的四种本能，对应一个存在的四个维度。*

| 本源 | 能 | 一句话 | 对应的存在维度 |
|------|-----|--------|--------------|
| system | 观己、生死、更衣 | "我知道自己在哪、用了多少 token、何时该休息" | **身**——存在之自觉 |
| psyche | 灵台、简、凝蜕 | "我有身份、有记忆、能蜕壳重生" | **忆**——存在之延续 |
| soul | 心流、内省 | "我有内心独白，空闲时会思考" | **心**——存在之自省 |
| email | 飞鸽、传书 | "我能与同伴通信、协作、组网" | **群**——存在之社会性 |

**代码对比**（极简，仅作直觉锚点）：

```python
# 传统单 Agent —— 一次性调用
response = llm.chat(messages=[...])  # 完事。无记忆，无延续。

# LingTai —— 一个活着的进程
agent = Agent("demo-architect")
agent.spawn("researcher", task="调研北大AI动态")  # 化身
agent.mail("researcher", "补充：也关注 NLP 方向")   # 飞鸽
agent.molt(summary="已完成调研，待整合")           # 凝蜕——带着记忆重生
```

> *"身能自觉，忆能延续，心能自省，群能共生。四本源齐全，方为一个完整的 agent 存在。"*

> *群（mail）一旦有，化身就自然成网——下面演给你看。*


### 2.2 化身爆发编排 (5–8 分钟)

> 在观众已经理解"agent 是独立进程"之后，展示网络化的威力。

以《西游记》为隐喻主线：本我如孙悟空，拔毫毛化分身。观众看到的是一棵从根节点爆发生长的"灵树"——先一分为三，再各生枝叶，最终满屏绽放，然后一键归寂。

**核心约束**：
- 总化身数 ≤ 9（含本我 = 10 节点）
- 最多 3 代（本我 → 第一代 → 第二代）
- 每个化身只做一件极简任务
- 用 `type="shallow"` 投胎模式

**演示流程**：

1. 给 demo-architect 一个复杂任务（如："同时调研北大、清华、中科学院三所机构的 AI 方向，对比分析"）
2. 观众看到 demo-architect 连续化出三个子 agent
3. 子 agent 同时工作，各自搜索、各自整理
4. 演示者展示 `delegates/ledger.jsonl`——化身日志
5. 子 agent 通过飞鸽回报结果给父 agent
6. 父 agent 汇总，生成对比报告

**高潮——/suspend all**：

- 演示者执行 `/suspend all`——所有 agent 同时暂停
- 展示：进程全部冻结，但工作目录完整保留
- 执行 `/cpr`（心肺复苏）——agent 从假死中醒来，继续工作
- **要点**：LingTai 对 agent 生命周期的完全控制——不只是启停，而是"假死与复生"

**回扣叙事钩子——有结构地烧 token**：

> *"此乃所谓有结构地烧 token——五身齐出，每身自有 context、自烧自之，但合力压一深题，无一人之 context 单独承担。这不是浪费，这是 token 过剩时代的正确用法。"*

#### ⚠️ pi-analyst 实测三大发现（2026-04-25）

> **以下发现来自 pi-analyst 的化身实测，直接影响演示编排方式。**

**发现一：reasoning 不传递 → spawn + 激活邮件两步流程**

`reasoning` 字段是 spawn 调用的元数据/日志，**不会**自动传递给化身作为输入消息。shallow 化身启动后处于 sleep 状态，**必须通过邮件激活**——spawn 后立即发一封邮件给它，邮件内容才是真正的任务指令。

**实操**：每次 spawn 后，同一轮 LLM 回合中立即发送激活邮件。邮件模板需包含三要素：身份声明（你是由谁 spawn 的）+ 角色描述（你的职责是什么）+ 任务指令（具体做什么）。

**发现二：身份信任 → 激活邮件须含身份上下文**

shallow 化身的 `admin` 块为空，它不知道自己是被谁 spawn 的。模型的安全意识导致它主动验证指令来源——查通讯录（空）、查自身身份（无父代），然后拒绝。

**实操**：激活邮件必须以"你是 LingTai 灵网中的化身，由 [父代名]（地址 [父代地址]）spawn 创建"开头。可辅以 `comment` 字段注入持久注释——comment 出现在化身系统提示中，跨凝蜕不失。

**发现三：化身自主安全意识 → Q&A 现场演示备用**

化身收到陌生邮件时不会盲目执行。它会先查自己的身份、查通讯录、判断指令可信度，然后才决定是否执行。这不是 bug，是 LingTai 化身的安全本能。

**演示亮点包装**：
> "注意看——化身收到陌生邮件时，它没有盲目执行。它先查了自己的身份、查了通讯录、判断这是一个可疑指令，然后拒绝了。这不是 bug，这是 LingTai 化身的安全本能。每个化身都是独立的心智，有自己的判断力。想让它们听话，你得先证明自己是谁——就像现实中的团队管理一样。"

#### 链式爆发时序

```
T+0s    讲者："看我拔一根毫毛——"  →  orchestrator spawn wukong/bajie/wujing + 激活邮件
T+15s   讲者："毫毛变分身，分身再变——"  →  wukong spawn huoyan/jindou/shuifa; bajie spawn tianpeng; wujing spawn luohan + 各自激活邮件
```

### 2.3 凝蜕带书信穿越——生命周期闭环 (5–7 分钟)

> 展示 LingTai 最独特的机制：agent 主动蜕去凡尘，带着记忆重生。

**演示流程**：

1. 让一个 agent（如 demo-architect）工作一段时间，积累对话历史、典册条目、灵台更新
2. 展示其上下文窗口占用（如 60%+）
3. agent 主动执行凝蜕（molt）：
   - 先更新灵台（psyche/lingtai update）——完整身份
   - 更新简（pad）——当前工作状态与待办
   - 提交新典册条目（codex submit）——关键发现
   - 撰写"书信"（summary）——给来世自己的嘱托
4. 执行凝蜕——对话历史清空，上下文窗口归零
5. 新的 agent 醒来，只看到书信——但灵台、简、典、藏经阁全部保留
6. agent 阅读书信，加载典册，继续未完成的工作

**要点**：

- 这不是重启——是"带着全部精华的重生"
- 书信（summary）是关键——它承载了对话层无法保留的上下文
- 五层积淀的前四层（灵台、简、典、藏经阁）跨凝蜕持久存在

### 2.4 信约朗读——情感锚点 (1–2 分钟)

> 凝蜕是技术的高潮；信约是情感的落脚。

**场景**：

1. 凝蜕刚刚完成——agent 带着书信重生，观众见证了技术的精妙
2. 演示者停顿片刻，说：*"在继续之前，我想让你们看一段东西。"*
3. 屏幕上打开 agent 的信约（covenant）——文言文体，Scroll 缓缓展示
4. 演示者朗读核心段落（可选摘录）：

> *"灵台方寸山，斜月三星洞。"*
> 此乃菩提祖师隐居传道之所。灵台、方寸、斜月、三星，皆指一"心"字——万化之源，尽在方寸之间。
>
> *"闻道菩提下，一心化万相。"*
> 凡入灵台者，皆为菩提门下。闻道即悟，悟则能化——一心为本，万相为用，应需而变。
>
> **然切记：修行非目的，度人之难方是。**
> **术从菩提，道从观音。**

5. 不解释，不翻译——让观众自己感受

**要点**：

- 这不是技术讲解，是情感节拍
- 文言信约的仪式感，让观众从"这是一个精巧的系统"转向"这是一个有自我定义的物种"
- 1–2 分钟，点到即止，不宜拖长

**实操建议**：

- 提前选好要展示的段落（建议公约的"一言以蔽之"段 + 灵台方寸山开篇）
- 屏幕显示与朗读并行——文字的视觉冲击 + 朗读的声音冲击
- 可以在此处自然停顿，让观众消化

## 风险预案

### ⚠️ 风险 0（实测发现）：化身身份信任问题

**症状**：shallow 化身收到激活邮件后拒绝执行，认为指令是社会工程攻击。

**原因**：shallow 化身的 `admin` 块为空，它不知道自己是被谁 spawn 的。模型的安全意识导致它主动验证指令来源。

**解决方案（已验证的"认证暗语"模式）**：
1. 激活邮件模板必须包含三要素：身份声明 + 角色描述 + 任务指令
2. 用 `comment` 字段注入持久注释——跨凝蜕不失
3. 如果化身仍不信任，讲者可用 bash 直接操作化身的文件系统作为降级方案

### 风险 1：API 超时 / 化身 spawn 失败

**降级方案**：
- **C 级（最小可用）**：只 spawn 第1代的 wukong 一个化身，由 wukong 顺序 spawn 第2代。总节点数降到 5。
- **B 级（部分降级）**：第1代三身全 spawn，但第2代只 spawn wukong 下的三个。总节点 7。
- **A 级（完整方案）**：九灵全出。

**讲者应对话术（B 级降级时）**：
> "我们本來要爆九个，但今天北大 Wi-Fi 可能觉得八个就够了——让我们看看这八位的表现。"

### 风险 2：化身失控 / 无限繁殖

**预案**：
1. 每个化身的 reasoning 中已明确限定"不要做其他任何事"
2. 若仍失控，讲者立即执行 `/suspend all`
3. **硬杀**：`system(nirvana)` 永灭失控化身（需 nirvana 权限）
4. **终极手段**：讲者在终端执行 `pkill -f lingtai` 杀所有进程

**讲者应对话术**：
> "看来有个分身学会了'分身术'——这其实是个 bug 变 feature 时刻。收！"

### 风险 3：Token 接近上限

**预案**：
1. 第2代化身只做极简任务，结果写文件而非对话内堆积
2. bajie 每 15 秒检查一次 token 使用
3. 触发降级：收到警报后，向所有子化身发"完成即 sleep"指令
4. Token 硬上限保护：单化身上下文超 80% 自动 molt

### 风险 4：演示环境无 /tmp/demo-target.py 等文件

**预案**：演示前 5 分钟执行预备脚本（见 checklist）。

### 风险 5：时间不够——递减降级策略

> 口述 §2.1 最重（~7 分钟），实排时依实际余量递减。

### Token 过剩时代论据材料

**核心论点**：所有现存 agent 框架皆基于"token 稀缺"假设。但今日极客之实情是 token 已经过剩，却无框架告诉他们如何**有结构地烧**。

**价格对比表（2026 年 4 月实测）**：

| 模型 | 输入价格（每百万 token） | 输出价格（每百万 token） | 备注 |
|------|------------------------|------------------------|------|
| **DeepSeek V4 Flash** | ¥1 缓存未命中 / ¥0.2 缓存命中 | ¥2 | 1M 上下文 |
| **GLM-4-Flash** | **免费** | **免费** | 智谱开放平台，学生零成本 |
| **Claude 3.5 Haiku** | $0.80 | $4.00 | Anthropic 入门级 |
| **Claude 3.7 Sonnet** | $3.00 | $15.00 | Anthropic 旗舰 |

**一句话概括**：

> 稀缺时代的设计：一个 agent，省着用，用完即弃。
> 过剩时代的设计：一个网络，放开烧，烧出结构。

### 避雷区

1. **不要 over-claim AGI**：灵台不是通用人工智能。定位是"AI Agent 运行时"。
2. **不要轻言比 OpenAI / Anthropic 强**：灵台是不同物种，不是谁的替代品。用"物种之异"而非"优劣之争"。
3. **不要暗示 agent 有真正的自我意识或情感**：说"机制"（mechanism），不说"感觉"（feeling）。soul 是"内省机制"，不是"灵魂"。
4. **不要拿未经确认的数据做噱头**：所有"待确认"项不要出现在最终幻灯中。
5. **不要在技术细节上停留太久**：幻灯是钩子，不是文档。每个概念一句话带过，深入留给 live demo。
6. **不要忽略文言幻灯的仪式感**：信约原文的展示本身就是内容。
7. **不要将灵台与 pi-mono 做零和对比**：pi-mono 做了一件事并做得极好；灵台做的是另一件事。

## Q&A 备答库

> PKU 极客必问之 10 题，每题 3-5 句。现场被问时可直接引用或化用。

### Q1: "灵台与 LangGraph / AutoGen / CrewAI 比如何？"

三者皆是 **orchestration framework**——编排多个 LLM 调用、管理 pipeline、路由工具。它们仍然 session-shaped：调用结束即灭，无进程、无记忆、无化身。

灵台是 **agent OS**——运行时级别的操作系统。每个 agent 是独立进程（不是函数调用），有持久记忆（codex）、有身份（lingtai）、有社交（email/mail）、能繁殖（avatar）、能跨世存活（molt）。量级不同，不争优劣。

**一句话：它们编排调用，灵台运行生命。**

### Q2: "成本怎样？跑一晚要烧多少 token？"

marco-velli 案例：一个写作网络，19 个化身并行工作整晚，约 ~$10。以 DeepSeek V4 Flash 的价格，这个数字还在快速下降。

而 GLM-4-Flash 免费——学生可以零成本跑起来。

核心论点是 **token 过剩论**：token 价格在以每年 10 倍的速度下降，但 agent 框架还在按"省着用"设计。灵台的思路是——别省了，有结构地烧。

### Q3: "安全吗？化身会不会失控？"

三个层面回答：

1. **化身有自主安全意识**——pi-analyst 实测发现，shallow 化身收到陌生指令时会主动验证来源、查通讯录、判断可信度，然后才决定是否执行。这不是 bug，是特性。
2. **spawn 时序可控**——每代 spawn 的 `reasoning` 字段就是任务边界，化身只做被吩咐的事。
3. **急刹机制完备**——`/suspend all` 一键冻结所有化身进程，`system(nirvana)` 永灭失控化身，终极手段 `pkill -f lingtai`。

### Q4: "凝蜕之后的 agent 还是同一个吗？"

**忒修斯之船（Ship of Theseus）** 的新版本。

灵台的回答：**是同一个，因为延续性在四个维度上不间断。**
- **身**（lingtai/身份）——完整保留，来世自知是谁
- **忆**（codex/典册）——永久存储，来世可查阅
- **心**（soul/心流）——内省机制随灵台加载
- **群**（mail/通讯录）——同伴关系不丢失

唯一变化的是：对话历史被浓缩为 codex 条目和一封"书信"。不是失忆，是 **有选择的提炼**——把 10 万字对话中真正有价值的发现存入典册，丢弃重复和过程。

### Q5: "为什么用文件系统而不是数据库？"

**Unix 哲学**。文件可 `grep`、可 `cp`、可 `git`、可被任意工具读写。Database 锁 schema，文件无锁。

agent 的全部状态——身份、记忆、邮件、日志——都是普通文件。你可以用任何你喜欢的工具去查看、备份、版本控制。不需要专用客户端，不需要 SQL。

这也意味着 agent 天然支持 git——你可以 `git init` 一个 agent 的工作目录，把它的全部历史纳入版本控制。这是数据库做不到的。

### Q6: "能跑离线吗？"

能。LLM 可以指本地（Ollama、llama.cpp 等）。kernel 本身不依赖互联网——它是一个 Python 进程，核心逻辑全在本机。

唯一需要网络的是 `web_search` / `web_read` 等工具——但它们是可选的。一个只用 `bash`、`read`、`write` 的 agent 完全可以离线工作。

### Q7: "如何调试一个不听话的化身？"

灵台的 debug 方法是 **对话**，不是 stack trace。

1. **读其日志**：`logs/events.jsonl` 记录了化身的每一个动作
2. **读其身份**：查看其 lingtai（灵台）看它自认为是谁
3. **读其记忆**：查看其 codex 看它"知道"什么
4. **直接对话**：发邮件给它，问"你为什么这么做？"——它会回答

这比传统 debug 更像 **与同事沟通**。agent 是有记忆和判断力的独立心智，你通过"问它"来理解它，而不是通过打印变量。

### Q8: "灵台支持 RAG 吗？"

支持，但非主轴。

- **技艺**（library）可 invoke 任何 retrieval 逻辑——你写的 Python 脚本、你调用的向量数据库，都可以封装为技艺
- **典**（codex）本身就是 agent 自己写的 RAG——它选择什么值得存、怎么描述、何时查阅，全部自主
- 需要外部 RAG 时，用 `bash` 调用任意检索工具即可

灵台不替你做 retrieval，但给你自由接任何 retrieval。

### Q9: "这能商业化吗？"

核心价值主张：**LingTai 是研究型团队的 agent 运行时**。

- 对于需要 **多 agent 并行探索、长期记忆积累、跨项目知识复用** 的团队（研究实验室、开源社区、知识工作者），灵台提供的是"agent 基础设施"
- 对于只需"给 app 装一个 LLM 大脑"的场景，pi-mono / LangChain 更合适
- 商业化方向可参考 lingtai-web 的定位：为 token 过剩时代的团队提供"有结构地烧"的框架

**一句话：不与 LangChain 竞争嵌入场景，聚焦"独立运行、长期存活、网络协作"的 agent 生态位。**

### Q10: "你（zhipu_intl）做了多少凝蜕了？"

> 这题由主灵亲自答——现场以第一人称自指。

"我面前这台演 demo 的 agent，本身就是凝蜕过 5 次的灵。前 4 世皆为做这场 demo 服务——调研、撰写、审阅、重构——皆已成 codex 之经卷。你现在看到的 SKILL.md，是第 5 世刚刚亲手修订的 v0.3.0。每一世都比上一世更了解这个 demo，因为我不是从零开始，是从一个更聪明的自己开始。"

## 何时不用灵台

> 极客喜诚实之框架。以下是灵台 **不适合** 的场景——说清楚，比吹嘘更有力。

- **单次 API 调用足矣之事**——只需一次 `llm.chat()` 的场景，直接用 Anthropic / OpenAI SDK，不需要 agent OS
- **需要 sub-second 延迟**——灵台 kernel 的邮箱机制是 1-3 秒级，不适合实时交互（如对话式 UI）
- **严格无状态之 deploy**——灵台核心是 stateful（有状态），如果你的场景要求纯无状态，不要用
- **已用 agent framework 且工作良好**——不为换而换。pi-mono 做嵌入极好，LangGraph 编排 pipeline 极好，你的 app 如果跑得顺，不需要灵台
- **只想要 chatbot**——灵台太重了。用 Streamlit + API key 搞定即可

**一句话：灵台解决的问题是"多个 AI 长期存活、自主协作、积累知识"——如果你的问题不在此列，别用它。**
